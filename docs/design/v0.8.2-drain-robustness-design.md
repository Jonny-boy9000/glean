## Problem Statement

v0.8.0/v0.8.1 shipped the drain core + UX polish. The drain *works* but isn't yet *trustworthy
unattended over a multi-day window*. Five gaps remain:

1. **Circuit breaker is a hard-coded `MAX_UNPRODUCTIVE = 3`** and only trips on literal zero-task
   bursts â€” a burst that "ran tasks but they all failed / were empty / instantly rate-limited"
   doesn't count as unproductive, so a degenerate window can spin.
2. **Mid-weekend re-discovery is incidental, not deliberate.** Each burst re-runs discovery, but
   there's no explicit guarantee that a multi-day window merges newly-appeared candidates and
   dedupes them correctly against work already done (stable `evidence_hash`), or that a file that
   *changed* under an in-flight draft is handled.
3. **Anti-spill is purely reactive.** A task can START seconds before a weekly reset and run into
   next week's fresh allowance. There's no pre-emptive margin.
4. **`today`/`peek` are single-run** while `morning` aggregates the whole drain window â€” so during a
   multi-day drain the three surfaces disagree ("morning says 4 branches, today says 1").
5. **The rate-limit classifier rests on an unverified guess** (ADR-0001). The signal that *ends a
   burst* â€” a real `claude -p` hard block â€” has never been captured.

## What Makes This Cool

This is the release that makes "open your laptop Monday and trust what the drain did all weekend"
real. A circuit breaker you can tune, a window that keeps finding fresh work without redoing old
work, a margin that refuses to spill into next week, three surfaces that finally agree â€” and a
classifier that's honest about what it knows, with a tripwire that captures the one missing piece
the first time it ever happens, instead of guessing.

## Constraints (inherited â€” non-negotiable, from CLAUDE.md)

Subscription-auth only (no API key in the core path); read-only against `main` (drafts to
`prep/glean-*` worktrees, never push/merge); deny-list on **every** `claude -p` spawn; default
`max_parallel=1`; Windows-first; cross-invocation drain state in a single atomic `state/budget.json`.
**The bare `glean run` path must stay byte-identical** (regression guard).

**Regression invariant (design rule for every lane):** all changes are **additive and gated on
drain state**. New `RunSummary`/config fields are optional and inert when unset; new behavior in
`today`/`pipeline`/`runDrain` only activates when a drain window is active (`readDrainState().kind
=== 'ok'` + a finite `drain_window_started_at`) or when `completedTaskIds`/drain-opts are passed.
With no window active, every code path the bare `glean run` touches is unchanged â€” same contract
already used by `pipeline.ts:47` (`completedTaskIds` inert) and `morning.ts` T6.

## Premises (confirmed this session)

1. The four robustness items + the reframed item 5 are the scope. No new candidate types, no API
   fallback (that's a separate roadmap item).
2. **Lane structure:** items 1â€“3 all converge on `runDrain.ts` + the pipeline execute loop â†’ one
   coordinated *drain-core* lane (sequential, TDD). Item 4 (`today`/`peek`) is fully independent â†’
   its own lane. Item 5 (`classify`/`executor` capture) is independent â†’ its own lane.
3. **ADR-0001 governs item 5.** `/usage` has no headless query (unchanged). `rate_limit_event`
   telemetry is real but every captured sample is a WARNING (`allowed`/`allowed_warning`) from a
   completed run; the hard-BLOCK shape was never captured. So: keep the stderr block detector
   load-bearing + unchanged; use `rate_limit_event` only for verified `resetsAt` (anti-spill) +
   `utilization` (telemetry); add a self-capturing tripwire. Do NOT make the event the primary
   classifier.
4. **Defaults (user-confirmed):** `anti_spill_margin_minutes = 15`; circuit-breaker threshold
   default `3` (was the hard-coded `MAX_UNPRODUCTIVE`), now configurable.

## Approaches Considered

### The rate-limit signal (the one real fork)
- **A (CHOSEN â€” per ADR-0001): enrich + capture, conservative.** Keep stderr block detection
  unchanged; parse `rate_limit_event` only for `resetsAt`/`utilization`; self-capturing tripwire.
  Effort S/M. Risk Low (executor streaming loop barely touched). Reuses the existing `.jsonl` capture.
- **B: proactive live pause on `allowed_warning`.** A's value plus the executor pre-emptively pauses
  at utilization â‰¥ 0.9. Effort M/L, Risk Med (edits the 789-LOC streaming loop). Deferred â€” nice,
  not load-bearing; revisit once a real block is captured.
- **C: fixtures + logging only, classify untouched.** Effort S, Risk Low, but leaves anti-spill
  without real `resetsAt` data. Rejected â€” `resetsAt` is verified and worth wiring now.

### Config surface (items 1, 3, 4)
- **CHOSEN: extend `drain_trigger` in `config.json`** with `max_unproductive?` and
  `anti_spill_margin_minutes?` (both optional, defaulted). Reuses the existing Zod schema +
  `DrainTrigger` type; no new config file. Threaded into `runDrain` via `PipelineOpts`/a drain-opts
  object so the state machine stays purely functional + testable with an injected clock.

## Recommended Approach â€” five lanes

**Lane D-core (items 1â€“3, sequential, one worktree â€” they share `runDrain.ts` + the execute loop):**

- **1. Configurable circuit breaker + richer "unproductive".**
  - `config.drain_trigger.max_unproductive` (default 3) replaces the `MAX_UNPRODUCTIVE` const,
    threaded through to `runDrain`'s guard 3d.
  - Broaden "unproductive". **Mechanism corrected (review):** `completed_evidence_hashes` is an
    *attempted* set, not a success set (`pipeline.ts:250` pushes for every executed candidate
    before the status switch), so it cannot be the lever. Instead add an explicit
    `productive: boolean` to `RunSummary`, computed in `finalize()`: **productive iff â‰¥1 task this
    burst produced non-trivial output** â€” i.e. `ran > 0` (a success; `ran++` only fires on the `else`
    branch at `pipeline.ts:260`) AND at least one of those outputs was non-trivial (reuse the
    renderer's existing trivial test: a draft with `files===0 || insertions+deletions===0`, or a
    dossier with no bytes, is NOT progress). Today's rule only caught `ran===0`; this also catches
    "ran but everything was empty."
  - **Reason â†’ unproductive mapping (explicit, review):** a burst counts toward the breaker ONLY
    when it genuinely tried and produced nothing. It does NOT count when: `reason` carries any
    rate-limit `classification` (session/weekly/ambiguous = a legit "come back later" wait â€” counting
    these would self-terminate a normal weekend, the v0.8.0 bug), or `reason âˆˆ {discovery-failed,
    lock-busy, not-eligible}` (transient/no-op). The repeated-pause case is already bounded by the
    separate `consecutive_ambiguous` + weekly/session guards, NOT this counter (no double-counting).
  - Files: `runDrain.ts` (guard 3d reads config threshold + `summary.productive`), `types.ts`
    (`RunSummary.productive?`), `config.ts` + `types.ts` (`DrainTrigger.max_unproductive?`),
    `pipeline.ts` (`finalize()` sets `productive`). Tests: threshold honored; `ran>0`-but-all-trivial
    trips; a real success resets; a rate-limit pause never trips; transient reasons never trip.

- **2. First-class mid-weekend re-discovery + `evidence_hash` dedup.**
  - Core guarantee (mostly hardening of existing behavior â€” `pipeline.ts` already re-discovers each
    burst and `runDrain.ts:100` passes `completedTaskIds`): make it **explicit + tested** that over a
    multi-day window, a candidate that first appears on day 2 is picked up, and a completed
    `evidence_hash` is skipped (`pipeline.ts:212`), so the window is never working off a stale day-1
    snapshot.
  - **Changed-file honesty (corrected, review).** `evidence_hash` for a TODO includes
    `todo_lines[].line` + `text` (`dedup.ts:stripVolatile` returns `todo` evidence unchanged). So a
    TODO whose **text** is unchanged keeps a stable hash (won't re-draft), but one that **shifts line
    number** (an edit above it) gets a *new* hash. We will **NOT** change `evidence_hash` globally to
    strip `line` â€” that would alter dedup for the bare `glean run` path (regression surface). Instead:
    document this as a known limitation, lean on the worktree/branch already existing + 21-day `gc`
    so a re-draft is at worst a second branch, not corruption, and **drop the absolute "changed file
    â†’ never re-drafted" claim**. Test the guarantee we actually have: identical-text TODO â†’ stable
    hash â†’ skipped.
  - Files: `pipeline.ts` (skip-set), `runDrain.ts` (skip-set union), `dedup.ts` (add a test pinning
    `todo` hash stability under identical text). Tests: new candidate mid-window â†’ picked up;
    completed `evidence_hash` â†’ skipped; identical-text TODO across bursts â†’ same hash â†’ not redone.

- **3. Anti-spill pre-emptive margin.**
  - **Placement decided (eng-review D1): PER-BURST, a new `runDrain` guard 3e** (after the
    no-progress guard 3d), NOT per-task in the pipeline loop. If `last_observed_weekly_reset` is
    known and `now()` is within `anti_spill_margin_minutes` of it (`reset - margin <= now < reset`),
    `runDrain` returns a no-op tick (`synthSummary('anti-spill')`) and waits for the next scheduled
    wake â€” consistent with guards 3a-3d. **`pipeline.ts` is untouched** (the bare-run execute loop
    never sees this), so item 3 is fully self-contained in `runDrain.ts` + config. Add `'anti-spill'`
    to `RunReason` (renders honestly via `outcomeLine`).
  - **Guard condition committed (review): use `now`** (not `now + est_task_time` â€” there is no
    `est_task_time` on `Candidate`, only `est_tokens`; and a 15-min margin already absorbs a task).
    Refuse to **start** a new task when `now` is within `anti_spill_margin_minutes` (default 15) of a
    **known** weekly reset.
  - **Reset source = `DrainState.last_observed_weekly_reset` ONLY for this lane (corrected, review).**
    That field exists (`state.ts:10`) and is read at the top of `runDrain` where the guard lives â€”
    fully self-contained in D-core. The richer `rate_limit_event.resetsAt` source is **NOT reachable
    here today** (it requires D-signal to parse `.jsonl` and thread `resetsAt` through) and D-signal
    is an independent lane â†’ wiring `resetsAt` into the margin is an explicit **follow-on integration
    step after both lanes merge**, not part of item 3's worktree.
  - **Document the genuine blind spot** (from the v0.8 design): on the very FIRST burst of a fresh
    window there is no reset estimate yet â†’ it stays reactive there. Correct, not a bug.
  - Files: `runDrain.ts` (guard reads `last_observed_weekly_reset` + the margin), config (margin).
    Tests: start refused inside margin; allowed outside; first-burst-no-estimate stays reactive.

**Lane D-surfaces (item 4, independent worktree):**

- **4. `today`/`peek` window-aware aggregation.**
  - **Shape mismatch resolved (review).** `today` returns a `TodayReport` (project-grouped
    `IndexEntry[]` parsed from INDEX.md + enriched from memory.db by `task_id`); `morning` returns a
    `MorningReport` (branches/files from `getRunsWithCandidatesSince`). These are different shapes, so
    `today` cannot literally call morning's path â€” it gets its **own window aggregator** that consumes
    `getRunsWithCandidatesSince(drain_window_started_at)` and emits a `TodayReport` (the same source
    of truth as morning, adapted to today's shape). Strictly **gated above the existing function** on
    an active window (`readDrainState().kind==='ok'` + finite `drain_window_started_at` + â‰¥1 window
    run); the no-window path is the *entire current function*, untouched â†’ byte-identical.
  - **`peek` must stay project-sliceable:** `findPeekDossier` filters by `project_slug` (`peek.ts:24`),
    so the aggregated `TodayReport` must keep per-project grouping (it does â€” `TodayReport.projects[]`).
    Test peek's slug filter against an aggregated window report.
  - Files: `today.ts` (new gated window aggregator â†’ `TodayReport`), `peek.ts` (inherits via
    `findTodayDossiers`; verify slice), reuse `memory.getRunsWithCandidatesSince` + `readDrainState`.
    Tests: during a window â†’ `today`/`peek` show the same branches+dossiers as `morning`; no window â†’
    byte-identical single-day output; peek still slices to the CWD project.

**Lane D-signal (item 5, independent worktree â€” bounded by ADR-0001):**

- **5. Honest signal: enrich + self-capture.**
  - `classify.ts`: keep stderr parsing; add an optional path to read `rate_limit_event` from a
    task's captured `.jsonl` and return `reset_at` from `resetsAt` (verified) when present. Stderr
    stays the block detector.
  - `executor.ts`: **self-capturing tripwire** â€” the first time a spawn is flagged `rateLimited`,
    dump the full raw stderr + last N stream messages to `~/glean/logs/<run>/<task>.BLOCK-CAPTURE.txt`
    so the missing block shape captures itself. Best-effort, never throws.
  - Commit the warning-shape fixtures (done) + tests against them; keep the ADR-0001 tripwire skip.
  - Files: `classify.ts`, `executor.ts`, `classify.test.ts`. Honors deny-list/auth/read-only (capture
    is local file write only).

## Success Criteria

- Circuit-breaker threshold reads from config (default 3, unchanged behavior when unset); a burst
  with `ran>0` but all-trivial output trips the richer unproductive rule; a real success resets it;
  a rate-limit pause and transient reasons (discovery-failed/lock-busy/not-eligible) never trip it.
- A multi-day window picks up a candidate that first appears on day 2 and never re-runs a completed
  `evidence_hash`; an identical-text TODO across bursts keeps a stable hash and is not redone. (A
  line-shifted TODO may yield a new candidate â€” documented limitation, bounded by worktree + gc.)
- A task is refused inside the 15-min pre-reset margin when `last_observed_weekly_reset` is known;
  first-burst (no estimate) stays reactive; nothing spills into the next week in a simulated boundary
  test.
- During a drain window, `today` and `peek` show the same branches/dossiers as `morning` (via
  today's own window aggregator over `getRunsWithCandidatesSince`); with no window active, their
  output is byte-identical to today's; `peek` still slices to the CWD project.
- `classify` returns the verified `resetsAt` from a real `rate_limit_event` fixture; the executor
  writes a BLOCK-CAPTURE file on the first rate-limit flag; the ADR-0001 tripwire remains skipped.
- **Bare `glean run` output + behavior unchanged** (regression). All existing 352 tests pass.
