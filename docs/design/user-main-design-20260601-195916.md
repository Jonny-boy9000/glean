## Problem Statement

Glean stops on the first rate-limit signal. Today the executor's `RATE_LIMIT_RE`
(executor.ts:13) only matches the *words* "rate limit / 5-hour / weekly", flips a
`rateLimited` bool, and the run ends — it captures no reset timestamp and never
classifies which limit it hit. That leaves most of a weekend's leftover weekly
subscription capacity unused, which is the exact thing glean exists to consume.

v0.8 makes glean drain the weekend's leftover capacity unattended: a scheduled
Thursday-evening start that works until the rolling 5-hour session window fills, exits
cleanly, gets re-launched after the window reopens, works again, and repeats across the
several 5-hour windows in the Thursday→weekly-reset tail — until the weekly cap is
exhausted or the reset arrives. Sunday/Monday morning the user opens their laptop to a
`glean morning` receipt (already shipped, v0.7.1) summarizing a *full* weekend drain
instead of a single burst.

It must do this while surviving the failure that killed the v0.7 in-process sleeper:
laptop lid-close / hibernate. The process must not hold a multi-hour timer; it exits and
is re-launched by Windows Task Scheduler.

## What Makes This Cool

"I closed my laptop Thursday with three days of unused Claude allowance. Sunday morning
I opened it and Glean had drained the whole weekend — several work bursts spread across the
rate-limit windows, four draft branches and two dossiers waiting, and my `main` was never
touched." A single overnight burst is a nice demo; a *fully drained weekend* is the
actual product promise the name "glean" makes. The `glean morning` receipt becomes the
payoff for the whole bucket, not a teaspoon of it.

## Constraints

(Inherited from CLAUDE.md load-bearing constraints — non-negotiable.)

- **Subscription auth, no API key.** Capacity consumed by spawning `claude -p`. No direct
  Anthropic API calls in the core path. (API-key fallback is a *separate* roadmap item.)
- **Read-only against the user's primary checkout.** Draft code lands on `prep/*`
  worktree branches under `~/glean/work/`. Never mutate main, never push, never merge.
- **Deny-list on every spawn** (hardened set from v0.7.0). Do not remove.
- **Default `max_parallel=1`.** Subscription sessions share one rate-limit bucket.
- **Windows-first.** Node + TypeScript; shell-out only for `git` / `gh` / `schtasks`.
- **Budget is indirect.** No headless usage query exists (`claude` has no `usage`
  subcommand; `/usage` is interactive-TUI-only). Glean reacts to stderr only; "drained
  the weekly bucket" is provable *only* when the weekly-limit stderr actually fires.

## Premises (confirmed in this session)

1. **Exit-and-re-enter, not an in-process sleeper.** The process dies cleanly on every
   rate-limit; Windows Task Scheduler re-launches it. (The sleeper died at lid-close and
   forked control flow on the known-buggy reset timestamp; rejected by the v0.7 eng review
   + an outside voice.)
2. **Two caps interact, and the 5-hour one is *why* re-launch exists.** The **weekly cap**
   is the big leftover bucket; the **5-hour session window** is a separate rolling throttle.
   You cannot burn the whole weekly bucket in one sitting — after ~5 hours of activity you
   hit the session wall and must wait ~5 hours for *that* window to roll over, even with
   weekly budget remaining. Draining a big bucket between Thursday evening and the weekly
   reset is therefore *several* work bursts (roughly three to six — depends on leftover size
   and the exact rolling-window mechanics, to be confirmed by the spike) separated by gaps of
   up to ~5 hours the process can't sleep through. Re-launch crosses those gaps. It is **not**
   "run every day all week."
   **EUREKA:** the re-launch machinery only does real work in the *high-leftover* case
   (you barely used Claude that week) — which is exactly the case with the most to drain.
   In the low-leftover case it drains in one burst, hits the weekly wall, stops, and the
   re-launch logic harmlessly no-ops. The feature's cost is paid only when its value is
   highest.
3. **One weekly Task Scheduler trigger, bounded to the drain weekend.** A single Scheduled
   Task starts Thursday ~18:00 local (configurable) and *repeats every ~60 min for ~60 h*
   — Task Scheduler's native repetition setting. Each repetition is an instant no-op while
   `now < next_eligible_at`; the one that lands just after a 5-hour window reopens runs the
   next burst. This matches the user's mental model (one weekly trigger) and handles the
   multi-window reality for free, without 24/7 polling.
4. **Resume / drain state lives in a single top-level `state/budget.json`**, NOT
   `candidates.json` (clobbered every run at pipeline.ts:151) and NOT a per-run file. It
   must persist *across* invocations. (See "§5.4 layout correction" below — the existing
   vision doc shows a per-run `budget.json`, which is wrong for cross-invocation state.)
5. **Classify rate-limit signals by reset *horizon* (hours vs days), not exact timestamp.**
   The displayed reset time is known-buggy and the weekly cap is a *rolling 7-day window
   anchored to the first prompt*, not a fixed calendar Saturday. Hours-away → session
   window, will reopen → set `next_eligible_at`, exit. Days-away → weekly cap exhausted →
   stop. **Never hardcode a calendar reset.** This is also why exit-and-re-enter is more
   robust than the sleeper on more than lid-close: a sleeper had to compute a precise sleep
   from the buggy timestamp; glean only needs "roughly when," and the scheduler heartbeat
   supplies the retry rhythm.
6. **One glean run spans exactly one weekly window.** Anti-spill stops glean from eating
   into the *fresh* new-week allowance once the reset passes. Next week is a separate
   scheduled invocation.

## Approaches Considered

### Re-launch mechanism

**Approach A: single weekly trigger + bounded intra-weekend repetition (CHOSEN).**
One Windows Scheduled Task, Thursday 18:00, repeat every ~60 min for ~60 h, "run ASAP
after a missed start" on. Re-entry guard no-ops early firings. Self-healing across crash,
sleep, and reboot — the exact failures that killed the sleeper. Effort: M. Risk: Low.

**Approach B: self-rescheduling one-shots (rejected).**
Each run schedules a one-shot Task at `next_eligible_at` and deletes itself. Precise, zero
idle wakeups — but if a run crashes *before* scheduling the next, the chain dies silently
and needs a watchdog anyway, plus more `schtasks` create/delete churn. The precision isn't
worth it when the reset timestamp is buggy.

**Approach C: hybrid heartbeat + precision one-shot (rejected for now).**
A reduces it: the repetition cadence already gives acceptable resume latency, so the second
scheduling path is complexity the validation doesn't need yet.

### Scope

**Approach A: split v0.8.0 / v0.8.1 (CHOSEN).** Core + scheduler first, validated on one
real weekend; robustness polish second, informed by observed behavior.

**Approach B: all-in-one v0.8 (rejected).** One big PR where a robustness-layer bug could
mask whether the *core* drain even works — the one thing v0.8 exists to prove — and where
the anti-spill margin and circuit-breaker thresholds get tuned against guesses instead of
observed real-weekend stderr timing.

## Recommended Approach

**A + split.** One weekly Task Scheduler trigger with bounded intra-weekend repetition,
shipped as v0.8.0 (core + scheduler) then v0.8.1 (robustness). Gets a real-weekend drain
datapoint fastest, matches glean's tiny-release cadence, and keeps each PR reviewable.

## Eng Review decisions (2026-06-02, /plan-eng-review)

Seven decisions resolved interactively against the actual code. These override the
implementation sketch below where they conflict.

- **D1 → per-burst runIds; `morning` aggregates the window.** Each burst keeps its own
  `runId` (faithful to N process invocations); `morning` gains a `getRunsSince(window)` query
  and merges all bursts since `drain_window_started_at` into one receipt. Without this the
  Monday receipt would show only the last burst (morning.ts:25 reads one run). `today`/`peek`
  window-consistency deferred to v0.8.1 (D7).
- **D2 → timestamp-scoped STOP.** The guard compares the STOP file's existing ISO timestamp
  (state.ts:67) to `drain_window_started_at`: a stop halts the current weekend; next
  Thursday's window ignores the stale stamp. `clearStop` is currently test-only (state.ts:70);
  without scoping, one `glean stop` silently kills every future weekend. `schedule disable`
  remains the deliberate permanent off.
- **D3 → atomic `budget.json` + corrupt-fails-safe.** Write via temp+`renameSync` (atomic
  replace on Windows). MISSING → fresh window; PRESENT-BUT-UNPARSEABLE → log + exit 0 without
  draining (never re-drain next week's bucket on a torn/corrupt read). Shared atomic-write
  helper reused for `summary.json`.
- **D4 → lock-free early-exit guard.** Read `budget.json` WITHOUT the lock at the top of the
  drain path; a no-op tick exits before lock/`recordRun`/sweep/gc/repair/discovery (so idle
  hours write nothing and don't pollute the aggregated receipt with empty run rows). Eligible
  → acquireLock → re-read under lock (authoritative; window-identity reset here) → burst.
- **D5 → thin `runDrain()` wrapper.** `runPipeline` stays the pure single-window "burst"
  (the bare `glean run` path provably unchanged), enriched only so a rate-limit result carries
  the classifier's `{kind, reset_at, horizon}` up into the summary. `runDrain()` does the guard
  + budget.json + interprets the summary. The `completed_task_ids` skip-set is injected INTO
  the burst as an opts field.
- **D6 → injected `now()` clock.** `runDrain`/guard take `now = () => Date.now()` so wake
  timing and the computed `next_eligible_at` are asserted exactly; the multi-invocation E2E
  advances a fake clock against a stub `claude`.
- **D7 → v0.8.1 TODO.** `today`/`peek` window-aware aggregation (match `morning`) is logged to
  ROADMAP v0.8.1, not built in v0.8.0.

**Folded (no decision):** classifier in a new pure `classify.ts`; centralized `reason →
exitCode` map for the new `RunReason` values; `schtasks` action uses the resolved ABSOLUTE
node+cli.js path and absolute `--project` (Task Scheduler has no PATH/cwd).

- API-key fallback on rate-limit — separate roadmap item; overlaps signal handling, sequence after.
- Configurable circuit-breaker threshold, first-class mid-weekend re-discovery, anti-spill

- `today`/`peek` window aggregation — v0.8.1 (D7).

## v0.8.0 — the drain core + scheduler

### 1. Signal classifier (executor) — the riskiest piece

> **CONTINGENT ON SPIKE 0 (usage-probe).** This entire stderr-reaction classifier exists
> only because there is no headless usage query (`/usage` is TUI-only; `claude -p` is
> headless, so it can't read it). Spike 0 (build-order step 1) re-verifies that as of today.
> If a scriptable usage query has since shipped, replace this section with a usage-poll
> (read remaining budget directly; sleep/exit when low) — the exit-and-re-enter + scheduler
> frame is unchanged, only the *signal source* flips from stderr to a poll. Build this
> classifier only after Spike 0 confirms no usage query exists.

Today `spawnClaude` returns `{ exitCode, rateLimited, timedOut, ... }` (executor.ts:667)
and `RATE_LIMIT_RE` (executor.ts:13) only matches words. Extend to:

```
classifyRateLimit(stderrText) → {
  kind: 'session' | 'weekly' | 'ambiguous',
  reset_at?: ISO string,        // parsed if present (treated as approximate)
  reset_horizon: 'hours' | 'days' | 'unknown'
}
```

**Horizon threshold (define it numerically — don't leave it to judgment):** parse the
reset moment; `reset_at - now < 6h` → `session`; `>= 6h` → `weekly`; no parseable reset →
`unknown`. The 6h cut sits just past the 5-hour session window so a full session reset
(≤~5h out) always classifies as `session` and a days-away weekly reset always as `weekly`.

- **`session`** → the 5-hour window is full; it will reopen. Set `next_eligible_at =
  reset_at` (or `now + 5h15m` if unparseable), persist, exit 0.
- **`weekly`** → weekly cap exhausted. This stderr is glean's *only* proof it drained the
  bucket. Set `week_exhausted = true`, record `last_observed_weekly_reset`, stop, report
  "drained weekly capacity."
- **`unknown` / ambiguous** → fail-safe. On the first occurrence, treat as `session` with a
  `now + 5h15m` retry; on a *second consecutive* `unknown` → **stop** and report "stopped
  early: ambiguous rate-limit signal." Never an unbounded re-entry loop.

**Degrade gracefully on a never-seen weekly string.** The spike (build order) can reliably
capture the *session* stderr but may not capture the *weekly* one on demand (see the spike
note). So the classifier must NOT depend on having seen the exact weekly wording: a parsed
reset `>= 6h` out classifies as `weekly` even if the surrounding words are unfamiliar. Ship
v0.8.0 with the session path fully tested and the weekly path horizon-driven + fixture-pending.

> **HARD DEPENDENCY (carried from v0.7's open questions):** the exact session-vs-weekly
> stderr wording and the reset-timestamp shape are **unknown**. Do **not** build the parser
> against assumed strings. **Spike:** capture raw stderr to fixtures and build + unit-test
> the classifier against those. **The two halves cost very differently:** the 5-hour limit
> is trippable in a few hours of activity; the *weekly* limit can only be captured
> opportunistically when you're genuinely near your cap (you cannot trip it in an hour
> unless you're already close). So design the classifier horizon-first (above) so it works
> on a never-yet-seen weekly string, and treat the weekly fixture as captured-when-available,
> not a build blocker.

### 2. Drain state: `state/budget.json` (single, top-level)

All timestamps stored and compared in **UTC** (render local only for display) — a DST
transition mid-weekend must not create an off-by-one-hour in the `next_eligible_at` compare.

```jsonc
{
  "drain_window_id": "2026-06-04T15:00:00Z",          // = drain_window_started_at, the window key
  "drain_window_started_at": "2026-06-04T15:00:00Z",
  "next_eligible_at": "2026-06-04T21:10:00Z",          // session reopens ~here
  "week_exhausted": false,
  "last_observed_weekly_reset": null,                  // set when a weekly signal fires
  "completed_task_ids": ["abc123", "def456"],          // resume cursor (bounded by candidates/window)
  "unproductive_reentries": 0,                          // v0.8.0 minimal no-progress backstop
  "schema": 1
}
```

Read/write helpers in `state.ts` next to the existing lock/STOP helpers. Corrupt/missing
file → treat as a fresh drain window (no crash) — an explicit success criterion.

### 3. Re-entry guard (top of `glean run --drain`, before discovery)

```
0. WINDOW IDENTITY. This invocation belongs to a NEW drain window if either:
   the budget.json is missing, OR now > last_observed_weekly_reset (the old week ended),
   OR now - drain_window_started_at > duration_hours (the repetition window elapsed).
   On a new window: reset week_exhausted=false, completed_task_ids=[],
   unproductive_reentries=0, stamp drain_window_started_at/drain_window_id = now.
   (This is what lets next Thursday's trigger start fresh — the code can't otherwise tell
   a fresh weekly trigger from a repetition heartbeat of the same task.)
1. If STOP sentinel present                          → exit 0 (let `glean stop` interrupt)
2. If week_exhausted AND now < last_observed_weekly_reset → exit 0 (bucket drained, wait)
3. If now < next_eligible_at                         → exit 0 (session window not reopened)
4. If unproductive_reentries >= MAX_UNPRODUCTIVE (3) → stop, report "stopped early:
   no progress" (the v0.8.0 backstop — see below)
5. Acquire RUN.lock; if busy → exit 0. Else proceed to a burst.
```

Idempotent and cheap — this is what makes the ~hourly heartbeat safe to fire constantly.
`RUN.lock` (state.ts:26, PID-liveness based) already prevents a heartbeat that fires
*during* an active burst from double-running; verify the busy path exits 0, not error.

**Stale-lock guard (reboot / PID recycle).** `acquireLock` trusts `process.kill(pid,0)`
liveness; after a reboot a recycled PID owned by an unrelated process would falsely read
"busy" and no-op every heartbeat for the rest of the weekend. The executor's per-task
timeout is 8 min, so a lock whose `started_at` is older than `task_timeout + margin`
(e.g. 20 min) with no fresh `summary.json` update is dead — treat it as stale and reclaim,
regardless of PID liveness. (Extend `acquireLock`'s recovery path, which already handles
corrupt locks at state.ts:38.)

### 3a. Minimal no-progress backstop (pulled into v0.8.0 — NOT deferred)

The full tunable circuit breaker is v0.8.1, but v0.8.0 needs a hard backstop, because the
*clean session-signal* path is itself an unbounded loop: a burst could instantly re-hit a
session limit before completing any task, persist a new `next_eligible_at`, exit 0 — and
repeat all weekend, terminating only on a *weekly* signal or STOP. So: after each burst,
if it completed **zero** new tasks, increment `unproductive_reentries`; if it completed
≥1, reset it to 0. Guard step 4 stops the window at `MAX_UNPRODUCTIVE = 3`. This is the
minimum that makes v0.8.0 safe to leave running unattended; v0.8.1 makes the threshold
configurable and adds the richer "all tasks failed/empty" classification.

### 4. Burst → exit (modify the pipeline.ts run loop)

A "burst" is the existing discover → dedup → rank → execute loop, with four changes:
- **Skip already-done work:** on re-entry, re-run discovery (cheap, read-only) and skip any
  candidate whose `task_id` is in `completed_task_ids`. Note `filterRecentlyProduced`
  (pipeline.ts:129) *already* suppresses recently-produced dossiers, giving partial
  idempotency for free; the completed-set covers draft-impl branches.
- **Discovery failure is non-fatal.** Over a multi-day unattended window, transient `gh`/network
  failures in `discoverGit` are near-certain. A failed discovery on re-entry skips the burst
  and exits 0 (retry next heartbeat) — it must NOT crash into the corrupt-state path.
- **On a `session` signal mid-burst:** persist `next_eligible_at` + `completed_task_ids`,
  update `unproductive_reentries`, write summary, **exit 0** (do not abandon, do not sleep).
- **On a `weekly` signal:** set `week_exhausted` + `last_observed_weekly_reset`, report
  drained, exit 0.

After every burst, update `unproductive_reentries` per §3a (zero new tasks → increment,
≥1 → reset).

### 5. Anti-spill (minimal in v0.8.0)

When a weekly signal fires, `week_exhausted` + `last_observed_weekly_reset` block any new
burst until that reset passes *and* a fresh weekly trigger fires. **First-task-blind
caveat (documented, not solved in v0.8.0):** at the very first burst of a fresh drain
window glean has no fresh reset estimate, so it cannot pre-emptively refuse a task that
starts seconds before a reset — the weekly stderr will catch it within one task. The
configurable pre-emptive margin is v0.8.1.

### 6. Honest reporting

`summary.json` + the `glean morning` capacity line:
- "drained weekly capacity" — **only** if a `weekly` stderr fired this drain window.
- "stopped early: <reason>" — for `budget` / `STOP` / `ambiguous-signal` / `no-progress` /
  `repetition-window-ended`. Never claim the bucket was drained without the weekly signal.
- **Wall-clock coverage line** — report how many bursts actually ran (e.g. "woke for 2
  bursts this window"). Critical because a laptop closed the *entire* weekend produces an
  empty receipt that must NOT be indistinguishable from a successful drain. "Drained weekly
  capacity" requires the weekly signal; "0 bursts — laptop asleep all window" is a distinct,
  honest outcome.

### 7. `glean schedule enable | disable | status`

A Windows Task Scheduler wrapper (`src/lib/schedule.ts`, Node `child_process`, no new deps).
Off by default (consistent with §5.5).

> **Feasibility reality (the review's biggest catch): `schtasks /create` flags CANNOT
> express the settings this design needs.** Repetition (`/RI` + `/DU`) is fine, but
> `StartWhenAvailable` ("run ASAP after a missed start"), `MultipleInstancesPolicy`,
> `DisallowStartIfOnBatteries`, and `StopIfGoingOnBatteries` are **not** settable via
> `schtasks` flags. So **enable generates a Task Scheduler XML** and registers it with
> `schtasks /create /XML <file> /TN Glean\Drain` (or PowerShell `Register-ScheduledTask` +
> `New-ScheduledTaskSettingsSet`). "A schtasks wrapper, no new deps" still holds — it's
> still just `child_process` — but the work is XML generation, not flag-passing. This is the
> single most underestimated piece of v0.8.0; budget for it.

- **enable** → generates and registers one Scheduled Task. Required `<Settings>`:
  - **Trigger:** weekly, **Thursday 18:00 local** (configurable via `config.json`
    `drain_trigger: { day, time, repeat_minutes: 60, duration_hours: 60 }`).
  - **Repetition:** every `repeat_minutes` for `duration_hours`.
  - `StartWhenAvailable = true` — run ASAP after a missed start (catch up on wake).
  - `<MultipleInstancesPolicy>IgnoreNew</MultipleInstancesPolicy>` — a firing during an
    active burst is dropped (belt-and-suspenders with `RUN.lock`).
  - `DisallowStartIfOnBatteries = false` **and** `StopIfGoingOnBatteries = false` — **without
    these the default Windows policy refuses to start (or kills mid-burst) on an unplugged
    laptop, silently defeating the entire weekend drain.** Mandatory, not optional.
  - `WakeToRun = false` — we *want* it to skip while the laptop sleeps and catch up on wake;
    that *is* the lid-close story.
  - `ExecutionTimeLimit` generous (a burst can be hours) or `PT0S` (unlimited), since the
    process self-terminates on a rate-limit signal anyway.
  - **Action:** `glean run --drain --project <configured path>` — **single project in
    v0.8.0.** (`--projects auto` interleaving across the shared rate-limit bucket is a
    separate Tracked "multi-project" item; do not couple it to the drain core.)
- **disable** → deletes the task. **status** → `schtasks /query /TN Glean\Drain` summary.
- **No hardcoded calendar reset anywhere in code** — the *trigger* is a user-chosen weekly
  time; the *stop* is driven entirely by stderr horizon.

> **`duration_hours` defaults to 60, not 36.** The weekly cap is a *rolling* window anchored
> to first-prompt, NOT a fixed Saturday — so a 36 h repetition window can end *before* the
> reset for some users, silently truncating the drain (the core promise quietly not holding).
> 60 h from a Thursday-evening trigger comfortably covers a Sunday-morning reset; widen
> further if real runs show truncation. The bounded window still avoids 24/7 polling.

> **AUTH FEASIBILITY — verify before anything else (could invalidate the whole mechanism).**
> `claude -p` consumes the user's **subscription session auth**, which lives in the user
> profile and is normally established in an *interactive* logon. A Task-Scheduler-launched
> process running "whether the user is logged on or not" runs in a **non-interactive** session
> and **may not find those credentials at all** — in which case the entire re-launch model is
> dead. The scheduled task therefore registers as **"run only when the user is logged on"**
> (so the real-world constraint is "laptop on and logged in, lid may close"), and the spike
> (build order step 1) MUST confirm a Task-Scheduler-launched `claude -p` authenticates
> before any engine work. If even logged-on non-interactive auth fails, the design needs
> rework — find that out in an hour, not after building the core.

### 8. `glean stop` interaction

With no in-process long sleep, STOP is simpler than the v0.7 sleeper needed: (a) the active
burst already checks `isStopRequested` between tasks (state.ts:63), and (b) the re-entry
guard's step 1 makes every subsequent heartbeat exit 0. So `glean stop` halts the drain
within one task + one heartbeat tick. Add the guard's STOP check; the burst-level check
already exists.

## v0.8.1 — robustness (deferred, informed by the first real run)

- **Configurable circuit breaker + richer classification.** v0.8.0 ships the minimal
  hard backstop (§3a: 3 zero-task re-entries → stop). v0.8.1 makes `MAX_UNPRODUCTIVE`
  configurable and upgrades "zero new tasks" to a richer "all tasks empty/failed/instantly
  rate-limited" signal, so a window that *runs* tasks which all fail also trips the breaker.
- **First-class mid-weekend candidate re-discovery.** A drain that spans Thursday→Saturday
  works off a Thursday snapshot. v0.8.0 already re-discovers per re-entry; v0.8.1 makes it
  deliberate — merge new TODOs/PRs that appeared mid-weekend, dedup against
  `completed_task_ids`, handle files that changed under it.
- **Anti-spill margin + pre-emptive refuse.** Add the configurable margin (the v0.7 guess
  was 15 min) and refuse to *start* a task within the margin of a *known* reset estimate.
  Tune against observed real-weekend stderr timing — which is why it waits for v0.8.0's run.

## Success Criteria

- **Classifier (mocked `claude` stub):** session-horizon stderr → records `next_eligible_at`,
  exits 0. Weekly-horizon stderr → `week_exhausted`, stops, reports "drained weekly
  capacity." Ambiguous/unparseable → fail-safe stop, never an unbounded re-entry loop.
- **Guard:** an invocation before `next_eligible_at` exits 0 immediately; one after it
  resumes the next candidate; a `week_exhausted` state no-ops until the recorded reset.
- **Resume:** completed `task_id`s are not re-run on re-entry; a corrupt/missing
  `budget.json` starts fresh without crashing.
- **No-progress backstop:** 3 consecutive zero-task bursts → stop, report "stopped early:
  no progress." A drain that re-enters into an instant session-limit wall does NOT loop all
  weekend.
- **Window identity:** after a `week_exhausted` weekend, an invocation past the recorded
  reset resets state and starts a fresh window (next Thursday's trigger isn't blocked by
  last week's `week_exhausted`).
- **Scheduler:** `glean schedule enable` creates a queryable Windows task with the required
  `<Settings>` — verify the generated XML actually sets `StartWhenAvailable`,
  `DisallowStartIfOnBatteries=false`, `StopIfGoingOnBatteries=false`, and
  `MultipleInstancesPolicy=IgnoreNew` (a `schtasks /query /XML` round-trip asserts them);
  `disable` removes it; `status` reports it. **No hardcoded reset time anywhere in code.**
- **STOP:** `glean stop` during a drain weekend → the next heartbeat's guard exits 0 with no
  burst; an in-flight burst stops between tasks.
- **Concurrency:** a heartbeat firing during an active burst sees the live PID via `RUN.lock`
  and exits 0 (busy), no double-run; a lock older than ~20 min with no fresh `summary.json`
  is reclaimed as stale (reboot / PID-recycle safety).
- **Coverage honesty:** a window where 0 bursts ran (laptop asleep) reports "0 bursts" and
  does NOT read as "drained weekly capacity."
- **Integration test (v21-style):** session-pause/resume, weekly-stop, ambiguous-failsafe,
  guard-noop-before-eligible, window-identity-reset, no-progress-backstop, STOP-interrupts-guard,
  lock-prevents-double-run, stale-lock-reclaim.
- **THE existential gate:** one **real** weekend drain (this repo or a real project)
  produces a Monday-morning `glean morning` receipt showing multiple bursts and an honest
  capacity line. v0.8.1 is gated on what this run reveals.
