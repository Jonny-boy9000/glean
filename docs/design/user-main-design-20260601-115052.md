## Problem Statement

Glean produces research dossiers and pre-fetched docs into `~/glean/` while a Claude
Pro/Max subscription sits idle at the tail of the weekly rate-limit window. It works,
but the output is quiet — a folder of markdown nobody screenshots. For an open-source
tool whose growth depends on word-of-mouth in the Claude Code / Max community, there is
no single moment that makes a dev say "whoa, I need this." The goal of this design is to
build that moment: idle subscription capacity that writes a reviewable branch overnight,
and a receipt surface that makes the result instantly shareable.

## What Makes This Cool

"I opened my laptop Monday morning and Glean had already written a branch for my top
TODO — +47/-3 across two files, six minutes of compute I'd otherwise have wasted, here's
the one command to review it." That is the screenshot. It is categorically more visceral
than "here is a research dossier," because a draft branch is *reviewable work*, not
reading material. The whoa lives in the receipt as much as the code: a narrated
"while you slept" summary turns an invisible overnight process into something you post.

## Constraints

(Inherited from CLAUDE.md load-bearing constraints — non-negotiable.)

- **Subscription auth, no API key.** Capacity consumed by spawning `claude -p`. No direct
  Anthropic API calls, no `ANTHROPIC_API_KEY` in the core path.
- **Read-only against the user's primary checkout.** All draft code lands on
  `git worktree`-based `prep/*` branches under `~/glean/work/`. Never mutate the main
  checkout, never push, never merge.
- **Deny-list on every spawn.** `--disallowedTools "Bash(git push:*) Bash(git checkout main:*) Bash(gh pr merge:*) Bash(gh pr create:*)"`. Do not remove.
- **Default `max_parallel=1`.** Subscription sessions share one rate-limit bucket.
- **Windows-first.** Node + TypeScript; shell-out only for `git`/`gh`. Worktree teardown
  must work through `jobobject.ts` child-tree-kill.
- **Base branch is explicit per-project** in `config.json` (`project_path → base_branch`).
  Missing entry → skip draft-impl for that project with a warning.

## Premises (all confirmed in session)

1. **draft-impl is a different value proposition than dossiers** — "review a branch" vs
   "read a doc." Current v0.3-v0.6 dossier telemetry will NOT predict draft-impl
   reception, so draft-impl is freed from the "wait for telemetry" gate in the roadmap.
   **EUREKA:** a flat dossier keep-rate most likely means "a doc isn't enough, I wanted
   code" — which argues *for* draft-impl, not against it. Gating the code feature on the
   doc feature's metrics measures the wrong thing.
2. **The demo GIF + one-command install (`npm i -g glean`) are part of THIS feature,
   not deferrable.** For the Claude Code crowd the whoa moment must be shown and the
   install must be frictionless, or the viral loop never starts.
3. **Worktree isolation + the deny-list ARE the trust story.** "It never touches your
   real branch, never pushes, never merges" is the pitch, and must be airtight and
   visible in the demo, not a footnote.
4. **This reuses existing plumbing** — `executor.ts`, `prioritize.ts`, `dedup.ts`, the
   `base_branch` config field, and the §5.3 worktree sketch. It's an executor addition
   (~600-800 LOC for the thin slice) plus a new template and a receipt renderer, not a
   greenfield build.

## Approaches Considered

### Approach A: Thin slice engine only
Add a `draft-impl` candidate type that takes the single highest-ranked TODO/FIXME,
provisions a worktree on `prep/glean-<slug>` off `base_branch`, spawns `claude -p` with
the deny-list, auto-commits to that branch. Review via `git checkout prep/...`. Fastest
to a working engine (~3-4 days), but the payoff is a bare `git checkout` — no demo.

### Approach B: Full sub-project per spec §5.3
Both code candidate types (draft-impl + draft-pr-reply), worktree gc/discard, structured
commits, INDEX apply/discard hints, plus durability (back-off ladder, resume-after-crash)
and distribution (npm publish, demo media). Best long-term architecture, but ~3-4 weeks
and you polish edges before one user has reacted.

### Approach C: Demo-first — thin-slice engine + "while you slept" receipt (CHOSEN)
Build A's thin-slice engine AND a `glean morning` surface that narrates the overnight
run. The receipt is the product surface; the code-gen is the backend. Front-loads the
viral artifact (~4-5 days).

## Recommended Approach: C — Demo-first

Chosen because the goal is open-source spread, and C is the only approach that directly
produces the artifact that spreads — the narrated overnight receipt — while keeping the
code-mutating engine to the safe thin slice. A ships a feature with no demo; B ships an
architecture nobody has reacted to. C ships the screenshot.

### Component breakdown

**0. FOUNDATION: weekly-window-aware budget model (~250-350 LOC) — build this FIRST**

Both dossiers and draft-impl sit on this. Today glean stops on the first rate-limit
signal, which wastes most of the weekly allowance and can kill a long draft-impl task
half-finished at the 5-hour wall. The goal: drain the week's leftover subscription
capacity, then stop cleanly without spilling into next week.

Hard facts (verified 2026-06-01 against Anthropic docs + claude-code issues):
- There is NO headless usage query. `claude` has no `usage` subcommand; `/usage` is
  interactive-TUI-only. So glean cannot read "weekly budget remaining" — it reacts to
  rate-limit stderr only. (Confirms the CLAUDE.md "budget is indirect" constraint.)
- The weekly cap is a **rolling 7-day window anchored to the first prompt**, NOT a fixed
  calendar time. The displayed "resets by" time drifts and has known display bugs. So
  glean must NEVER hardcode a reset time (e.g. "Saturday 11:00 AM"); it reads the reset
  moment from the error text and treats it as approximate.
- The 5-hour "current session" window is also rolling from first request, separate from
  the weekly cap.

Logic:
- **Classify every rate-limit signal by its reset horizon** (parse the reset timestamp
  from stderr):
  - reset is *hours* away → 5-hour session window is full → **pause**, sleep until just
    after the reset, **resume** the run. Keep doing this to drain weekly capacity across
    successive 5-hour windows.
  - reset is *days* away → **weekly cap exhausted** → stop. This stderr is glean's only
    proof it drained the bucket ("used nearly all of it" is not otherwise measurable).
- **Resume-after-pause** (promotes the deferred backlog item to load-bearing): persist
  partial run state to `candidates.json` so a run continues after a session-window sleep
  rather than abandoning it. This is also what lets a long draft-impl task survive a
  5-hour rollover mid-branch.
- **Anti-spill margin:** refuse to start a new task within a configurable margin
  (default 15 min) of the most recently observed weekly-reset timestamp (persisted to
  state). One `glean` run spans exactly one weekly window; next week is a *separate*
  scheduled invocation, never an auto-continuation across the boundary.
- **Honest reporting:** the receipt says "drained weekly capacity (hit weekly limit)"
  only when the weekly-limit stderr actually fired; otherwise "stopped early: <reason>"
  (budget, STOP sentinel, anti-spill margin). Never claim "used all your usage" without
  the weekly signal.

Scope cut for v0.7: the pause/resume/classify/anti-spill *core* is in. Full unattended
all-week automation (a scheduler that wakes glean every ~5 hours for days without the
user present) is **v0.8** — it needs Windows Task Scheduler integration (the deferred
Scheduling sub-project).

**1. Engine: `draft-impl` candidate type (~700-900 LOC — this is a real feature slice,
not a trivial add; the "thin" refers to single-candidate scope, not LOC)**
- New candidate type alongside `research-dossier` / `fetch-docs`. Prioritizer type weight
  `draft-impl = 1.0` (already specified in §5.2). **Schema:** extend the `IndexEntry.type`
  closed union (currently `'research-dossier' | 'fetch-docs'`) to include `'draft-impl'`,
  and add diff-stat fields (files-changed, insertions, deletions, prep-branch name) to the
  run record / memory.db so the receipt can read them. `today.ts` joins on `task_id` — keep
  that join key intact for draft-impl rows.
- Workspace provisioning: `git -C <main> worktree add ~/glean/work/<slug>-<taskid> -b prep/glean-<taskid> <base_branch>`. Guard: if `base_branch` missing from config → skip with
  warning (premise 4 / existing decision). Pin `<base_branch>` to the exact ref the worktree
  is created from (local `main`, not `origin/main`) so the later diff is honest.
- New template `templates/draft-impl.md`: instructs the spawned session to implement the
  one TODO/FIXME, run the project's own tests if present, and commit. Deny-list passed
  always — **see hardened deny-list under Trust story.**
- **Spawn permissions (failure mode the reviewer caught):** the draft-impl session is the
  first Glean path that actually runs `Bash` (git commit, test runner). The dossier path
  never did, so `--permission-mode acceptEdits` has never been exercised against Bash. A
  headless `-p` run that needs `git commit`/`npm test` must NOT hang on an interactive Bash
  approval. Pass explicit `--allowedTools` for the git/test commands the template uses, and
  verify non-interactive Bash before wiring the loop. This is the most likely silent
  overnight failure; test it first.
- **Commit is best-effort, not guaranteed.** The session may edit but never commit (no green
  tests, budget exhausted, just didn't). After the session exits, executor checks the
  worktree for uncommitted changes: if dirty, auto-commit them with a `glean: draft (review)`
  message so the diff is never silently empty; if clean and no commit exists, mark the
  candidate failed and keep the worktree for inspection.
- **Diff capture (exact form):** `git -C <main> diff --stat <base_branch>...prep/glean-<taskid>`.
  Run with `-C <main>` (or any dir in the shared repo) — all linked worktrees share one
  object store, so this reads correctly on Windows regardless of where the worktree dir
  lives. Never run a bare `git diff` inside the worktree (that's working-tree vs index, not
  the branch delta).
- **Tests-before-commit (resolved open question):** attempt the project's tests, commit
  regardless, and surface pass/fail prominently in the receipt. Rationale: an honest "draft,
  tests red" branch is still reviewable work and better than discarding it; the receipt's
  test-status line keeps trust intact. The launch GIF simply uses a TODO whose draft lands
  green.
- INDEX.md gets a `draft-impl` entry with the (corrected) review command and a discard
  command — both defined under Component 2.

**2. Surface: `glean morning` (~250-350 LOC)**
- Extends the `today.ts` / `peek.ts` renderer. For the most recent run, prints a
  per-candidate "while you slept" block:
  - branch name, diff stat (`+47 / -3`, 2 files), files touched
  - what it attempted (the TODO text / candidate title)
  - test status (pass/fail/none), minutes spent, rate-limit hits, capacity note
  - **the review command (corrected — the naive `git checkout prep/...` in the main repo
    FAILS):** because the prep branch is already checked out in the linked worktree, the
    main repo cannot check it out (`fatal: 'prep/...' is already used by worktree at ...`).
    Two correct paths, both shown:
    - inspect in place: `cd ~/glean/work/<slug>-<taskid>`
    - read-only diff without moving: `git -C <main> diff <base_branch>...prep/glean-<taskid>`
  - **discard command:** `git -C <main> worktree remove --force ~/glean/work/<slug>-<taskid> && git -C <main> branch -D prep/glean-<taskid>` (a plain `rm -rf` leaves a dangling worktree
    registration — must go through `git worktree remove`). `glean gc` enumerates via
    `git -C <main> worktree list` and applies the same removal to entries older than 21 days.
  - Apply / Discard hints
- Works for dossiers AND draft branches — unifies the output story so every Glean run has
  one legible morning summary.
- ANSI-colored, screenshot-able. This is the demo frame.

**3. Distribution (premise 2 — but gated on one real run, per reviewer's judgment flag)**
The roadmap defers npm/GIF/CI "until telemetry validates the core" in three places. Premise
1's EUREKA (dossier metrics can't predict draft-impl reception) is plausible but asserted,
not evidenced — and the cross-model cold read was skipped this session. So: do NOT publish
on faith. Sequence it:
- Build the engine + `glean morning`. Run it for real against this repo for one night.
- If that single overnight run produces a branch *you* would actually review → record the
  ~20s terminal GIF (`glean run` → `glean morning` → `cd` into worktree → diff) and drop it
  into the README `<!-- TODO -->` placeholders. The GIF is the conversion lever; it's
  earned by one real result, not deferred indefinitely.
- `npm publish` (manual is fine for first ship). **Defer** the GitHub Actions npm-on-tag
  release job — that's YAGNI until the feature has produced more than one branch.

### Trust story (premise 3 — must be visible AND airtight)
The `glean morning` receipt explicitly states "branch `prep/glean-<taskid>` — isolated
worktree, your `main` was never touched, nothing was pushed." The deny-list and worktree
isolation become a *named selling point* in the README and the demo caption, not an
implementation detail.

**Hardened deny-list (reviewer caught a real hole).** The current four blocked prefixes
(`git push`, `git checkout main`, `gh pr merge`, `gh pr create`) are bypassable: `git switch main`
sidesteps the `git checkout main` block entirely, and `git branch -f` / `git reset` can move
refs. For the draft-impl path, extend the deny-list with `Bash(git switch:*)`,
`Bash(git branch:*)`, `Bash(git reset:*)`, `Bash(git worktree:*)`. **Defense in depth:**
isolation is the real guarantee — `main` is checked out in the user's primary worktree and
git itself refuses to let a linked worktree mutate another worktree's HEAD. The deny-list is
the second layer, not the only one. Verify both: write a test that a draft-impl session
cannot alter the main worktree's HEAD even if its template tries.

- **Exact rate-limit stderr format must be captured empirically.** The drain core's
  signal classifier depends on the precise text/shape of the 5-hour vs weekly limit
  messages from `claude -p` (the reset-timestamp format, the wording difference between
  the two). Before building Component 0, deliberately trip both limits once and record
  the raw stderr — do not build the parser against assumed strings. The displayed reset
  time is known-buggy, so the classifier should lean on the reset *horizon* (hours vs

  timing is observed.

## Success Criteria

- **Drain core:** against a mocked `claude` stub that emits a 5-hour-limit stderr,
  glean pauses and resumes (does not abandon the run); against a stub that emits a
  weekly-limit stderr, glean stops and reports "drained weekly capacity." A stub whose
  weekly-reset timestamp is within the anti-spill margin causes glean to refuse new tasks.
  No hardcoded reset time anywhere in the code.
- `glean run` against a project with a `base_branch` and a real TODO produces a committed
  `prep/glean-<taskid>` branch whose diff implements that TODO, with `main` untouched.
- `glean morning` renders a screenshot-able receipt for that branch, including the review
  command, the "never touched main" trust line, and an honest capacity line (drained vs
  stopped-early).
- The README carries a real ~20s GIF of the loop, and `npm i -g glean` installs a working
  binary.
- Integration test (v16-style) asserts: worktree created on the right base, commit exists
  on the prep branch, main HEAD unchanged even if the template attempts `git switch main`,
  receipt line matches the diff stat.
