## Problem Statement

Three observations from the founder, one from the field:

1. "Glean isn't able to properly use up all of the leftover usage in time for the
   weekly reset." Verified: the 2026-06-11 drain ended with `week_exhausted: false`
   and two no-op re-entries. It ran out of WORK, not capacity.
2. Glean should also run nightly — prep for tomorrow morning, gated by how the
   user's actual weekly usage compares to their own normal pace.
3. Glean should pick the right Claude model per task, and the founder asks whether
   a central orchestrator agent should spawn subagents.
4. Field evidence (2026-06-11 logs): 7 of 16 tasks died on the 5-hour 429 and were
   wrongly recorded as completed; the orchestrator kept spawning doomed tasks after
   the first 429. Both engine bugs are being fixed on `fix/drain-rate-limit-resilience`.

## What the data says (evidence, not vibes)

- **Why the week's capacity goes unused — ranked root causes:**
  1. **Candidate supply** is the binding constraint: glean-on-glean produced 16
     candidates; 9 ran in ~42 min of one 5-hour window. The remaining ~59 hours of
     the drain window had nothing to do. More repos + deeper task types is the fix,
     not more drain mechanics.
  2. The **failed=completed bug** deleted 7 retryable tasks (44% of the queue).
  3. **No second-pass depth**: a finished dossier never spawns follow-on work
     (e.g. dossier → draft-impl on its top recommendation).
  4. Serial, 8-min-capped tasks bound consumption per window.
- **What the telemetry can and cannot see** (verified against 88 captured events):
  `rate_limit_event.rate_limit_info` carries `utilization` (0→1), `rateLimitType`,
  `resetsAt`, `status`, `surpassedThreshold` — but **only `five_hour` has ever been
  observed**. The weekly window is invisible until (presumably) it trips. Therefore:
  - Real-time governing of the ACTIVE session window: **possible today.**
  - Absolute weekly-quota tracking: **impossible** under subscription auth (ADR-0001).
  - Weekly pacing must be **self-relative**: local JSONL token accounting
    (`~/.claude/projects/**/*.jsonl` usage fields — the ccusage approach) compared
    against the user's own trailing multi-week baseline. This is exactly the
    founder's framing: "in relation to your normal usage."

## Premises (challenged, then accepted)

1. **The right problem is supply + pacing, not parallelism.** Parallelism without
   more candidates just exhausts a small queue faster. Agree.
2. **Self-relative pacing is honest; absolute quota tracking would be a lie.**
   Any UI claiming "X% of weekly quota left" would be invented. We show "vs your
   normal week" instead. Agree.
3. **No in-process multi-hour sleeping** (prior learning, confidence 9/10:
   laptops hibernate at lid-close). Nightly mode = more trigger times for the same
   exit-and-re-enter machine, never a resident daemon. Agree.
4. **Deterministic orchestration stays code, not an LLM.** An orchestrator agent
   would spend the very capacity it manages on coordination, non-deterministically.
   Rejected as architecture; accepted as a bounded, optional triage pass (below).

## The Orchestrator Question (founder input, answered)

**Should a central orchestrator agent spin off subagents? No — with two yeses.**

- **No resident orchestrator agent.** Discovery, ranking, budgeting, retries and
  pacing are bookkeeping. Code does this deterministically, testably, and for free.
  Every token an orchestrator-agent burns is a token not spent producing output,
  and its failures (hallucinated priorities, drifting prompts) are exactly the
  failures glean's ledger/dedup design exists to prevent.
- **Yes #1 — in-task fan-out.** A headless `claude -p` session already has its own
  agent tool. The dossier/draft templates can instruct: "for breadth, dispatch
  parallel subagents." glean gets multi-agent parallelism inside one task with
  zero new process management — the session manages its own swarm within the same
  rate bucket.
- **Yes #2 — a bounded triage pass.** CLAUDE.md anticipated upgrading heuristic
  ranking "if rankings prove bad post-MVP." They have (stale titles, a blank
  title, a false-positive TODO got rank 1). One cheap-model `claude -p` call per
  drain window re-scores `candidates.json` (est_value, est_tokens, stale?, dup-of?)
  for a few cents of capacity. Flag-gated: `--triage`.

## Model routing (founder input, answered)

`claude -p --model <alias>` is the lever. Two rules compose:

- **Task-type default**: `fetch-docs → haiku`, `research-dossier → sonnet`,
  `draft-impl → default/opus` (reviewed code deserves the strongest model).
- **Pacing override — route UP when draining, DOWN when conserving.** Leftover
  end-of-week capacity has zero marginal cost; the weekend drain should use the
  most capable model available. A nightly run during an over-pace week drops to
  haiku or skips. Config:
  ```json
  "models": { "fetch-docs": "haiku", "research-dossier": "sonnet", "draft-impl": "opus" },
  "pacing":  { "over": "haiku-or-skip", "under": "promote-one-tier" }
  ```
- **Research findings (verified 2026-06-12):** `--model` is documented for `-p`
  with aliases `sonnet|opus|haiku|fable` + full ids; `--fallback-model` exists but
  explicitly does NOT trigger on rate-limit errors (can't dodge the cap with it).
  Anthropic's help center: "Opus costs several times more per turn than Sonnet,
  and Sonnet more than Haiku" — no published numeric multiplier. **Since Nov 2025
  Max plans carry TWO weekly limits: all-models + a separate Sonnet-only pool.**
  Max accounts default to Opus, so an unrouted glean drain burns the most
  expensive capacity available. **Drain on `--model sonnet` by default** — it
  consumes the shared cap several times slower AND draws from a pool that
  otherwise goes entirely unused. Record the pool-split as a new ADR assumption
  (whether Pro also splits is unverified) and confirm empirically via the
  `rateLimitType` values glean already logs.

## Approaches Considered

### Approach A: Pacing gate only (minimal viable)
Summary: `glean usage` (local JSONL accounting + baseline + pace ratio) and a
nightly trigger preset whose only smartness is skip/small/normal/large budget tiers.
Effort: S–M. Risk: Low. Reuses: discover-jsonl's JSONL access, schedule.ts, runDrain.
Pros: ships in days; immediately answers "should glean run tonight?"; no new
process model. Cons: leaves the live-window governor and supply problem untouched.

### Approach B: Full capacity governor, staged (ideal architecture) ← RECOMMENDED
Summary: A as v0.9.0, then: utilization-aware admission control (skip tasks whose
estimate exceeds remaining 5h-window headroom; generalizes the last-30-min rule),
adaptive parallelism (raise --parallel near week-end when utilization is low),
actual-vs-estimate token calibration recorded in memory.db, model routing, triage
pass, dossier→draft-impl follow-on chaining (fixes supply depth), multi-project
discovery emphasis (fixes supply breadth).
Effort: L total, but every stage ships independently. Risk: Med (estimation
quality), mitigated by calibration loop. Reuses: classify.ts horizons,
rate_limit_event capture, est_tokens, memory.db schema, prioritize.ts.

### Approach C: Orchestrator-agent architecture (lateral — rejected)
Summary: a resident `claude -p` orchestrator plans and spawns workers.
Why rejected: burns governed capacity on coordination; duplicates ledger/dedup in
prompts; unverifiable behavior vs. glean's 436-test deterministic core; the two
"yes" slices above capture its value at ~2% of its cost.

## Recommended Approach

**B, staged.** v0.9.0 = Approach A (usage + pacing + nightly preset) plus the two
in-flight bug fixes **plus two one-line executor wins from research: `--model
sonnet` default on drained tasks (taps the separate Sonnet weekly pool) and
`--max-turns` on every spawn (runaway-loop guard)**; v0.9.x adds admission
control + calibration + full model routing; v0.10 adds triage + follow-on
chaining. Positioning shift: glean stops being "a
weekend drainer" and becomes **the thing that guarantees a Claude subscription is
never wasted** — nightly prep when you're under pace, full drain at week-end,
honest receipts always.

## Ecosystem / integrations (research verified 2026-06-12)

- **ccusage (MIT)** reads the same `~/.claude/projects/**/*.jsonl` glean already
  walks, sums the per-message `usage` blocks, and exposes BOTH `--json` CLI output
  (incl. `blocks --json` modeling 5-hour windows) and a programmatic
  `ccusage/data-loader` API. Its issue tracker (#888, #866) documents real dedup
  undercounting — local accounting is an estimate, not ground truth. Decision:
  **consume ccusage as a dependency or shell-out for `glean usage`** rather than
  re-owning the dedup bug surface; pin the version.
- **Two stronger-but-unofficial weekly signals exist**, both known-not-adopted:
  the undocumented OAuth `api.anthropic.com/api/oauth/usage` endpoint
  (authoritative weekly %, fragile, User-Agent-gated) and the statusline stdin
  `rate_limits.seven_day` (Claude Code ≥2.1.80). If ever adopted (e.g. pre-drain
  headroom check), gate behind a new ADR + feature flag; the in-stream
  `rate_limit_event` stays the source of truth.
- **Anthropic now ships native scheduling** (session crons, Desktop scheduled
  tasks, cloud Routines — Pro 5/day, Max 15/day). None of it does glean's job:
  Routines are fresh-clone-only (no local JSONL/git context), Desktop tasks don't
  budget-drain. Positioning: glean's moat is *discovering what's worth doing* from
  local history; launch docs should say so explicitly.
- **Community overnight runners** (ClaudeNightsWatch, claude-code-schedule,
  night-watch-cli, AutoClaude) all require the user to pre-write the task — none
  discover work. Confirms the differentiator; nothing worth depending on.
- **Hard-won community lesson glean should adopt today: pass `--max-turns` on
  every spawned session** (documented print-mode flag) as a runaway-loop guard
  orthogonal to the 8-min timeout. (`--max-budget-usd` caps API dollars — a no-op
  under subscription auth; don't rely on it.)
- **MCP dossier server** (`glean mcp`, stdio search/read over `~/glean/dossiers`)
  — the proven Basic-Memory/Obsidian-MCP pattern; promotes dossiers from "folder
  you forget to open" to "context that shows up in any session." Promote from
  deferred backlog to v1.0 candidate. Zero-code interim: one README paragraph on
  pointing Basic Memory/Obsidian at `~/glean/`.
- **Linux substrate verified**: systemd user timers with `Persistent=true` (+
  `loginctl enable-linger`) are the only user-level mechanism that fires missed
  hourly ticks on wake — 1:1 with Task Scheduler's `StartWhenAvailable`. Cron is
  the degraded fallback only. An always-on Linux box is arguably glean's best
  home — no lid-close problem at all.

## Success Criteria

1. A scheduled weekend drain ends with `week_exhausted: true` OR an empty candidate
   queue across ≥2 configured projects — and the receipt says which.
2. Nightly mode runs ≥5 consecutive nights unattended, choosing tier by pace, with
   zero double-runs (ledger) and zero runs during over-pace weeks.
3. Failed tasks retry on the next tick; a 429 mid-run stops the burst within one task.
4. `glean usage` matches manual JSONL accounting within 2%.
5. Dashboard shows live five_hour utilization and pace ratio.
