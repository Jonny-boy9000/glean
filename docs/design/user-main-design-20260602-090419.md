## Problem Statement

v0.8.0 (the drain core) shipped and published. Three rough edges remain that affect whether
the drain feels good to actually use and show off:

1. **The scheduler default day is hardcoded Thursday 18:00** — correct for the author (Israel,
   Sun–Thu work week) but wrong for the Mon–Fri majority, who'd want Friday evening. The drain
   should fire at the end of the *last work day*, when weekly Claude capacity is most idle.
2. **A run's outcome is terminal-only and ephemeral.** `glean morning` renders ANSI text
   (render-morning.ts); to share the "look what it drained over the weekend" moment you
   screenshot a terminal, and it's gone when the window closes. There's no at-a-glance header
   (which project, how much drained, totals).
3. **The GitHub README is stuck at MVP/v0.1 era** — it still calls draft-impl and scheduling
   "future," tells users to `git clone` instead of `npm i`, and its FAQ contradicts shipped
   behavior (drain pause/resume, draft-impl, `glean schedule`).

## What Makes This Cool

The drain already produces real work over a weekend. These three turn "it ran" into "look
what it did" — a Monday-morning `RECEIPT.md` you can paste straight into a PR, Slack, or a
tweet (project + capacity drained + N branches/dossiers + diff totals, then the detail), a
scheduler that's correct out of the box wherever you are, and a README that actually matches
the tool you'd install today.

## Constraints

(Inherited — non-negotiable.) Subscription-auth only; read-only against `main`; deny-list on
every spawn; Windows-first; no new heavy runtime deps. Presentability must not regress the
bare `glean run` / `glean today` output.

## Premises (confirmed this session)

1. **The trigger is about the WORK WEEK, not the weekly reset.** The weekly cap is
   rolling-from-first-prompt (v0.8.0), so the right default is "end of your last work day"
   (Israel: Thursday; Mon–Fri: Friday), not a fixed reset clock.
2. **Markdown is the right shareable form** for v0.8.1 — renders cleanly on GitHub/Slack,
   screenshots well, reuses the existing renderer's PLAIN painter, and needs zero new deps.
   An image share-card (HTML→PNG) is deferred (would pull in a rendering dependency).
3. **The README refresh is mechanical once 1 & 2 are decided** — it must document the new
   scheduling default and the receipt artifact, so it sequences last.

## Approaches Considered (decided)

### Item 1 — scheduler default day
- **A (CHOSEN): timezone auto-detect + announce + override.** `Asia/Jerusalem` (and other
  Sun–Thu zones) → Thursday; else Friday. Print the choice; `drain_trigger.day` / `--day` win.
- B: hardcode Friday + document Israel override (wrong-by-default for the author).
- C: prompt at `schedule enable` (breaks non-interactive enable).

### Item 2 — presentability
- **A (CHOSEN): durable RECEIPT.md + summary header + totals**, terminal output unchanged.
- B: terminal-only polish (still ephemeral).
- C: HTML→PNG share-card (deferred — rendering dependency).

## Recommended Approach (v0.8.1 = three components)

### Component 1 — work-week-aware scheduler default
- New pure helper `defaultTriggerDay(tz = Intl.DateTimeFormat().resolvedOptions().timeZone): 'Thursday' | 'Friday'`.
  - A small lookup set of Sun–Thu work-week timezones → Thursday. Start with `Asia/Jerusalem`
    only (the known case); everything else (incl. `'UTC'` / unknown) → Friday. Keep the set
    tiny + commented. `tz` is a default param evaluated at CALL time → unit-testable by passing
    a fixed tz, no `Intl` mocking, no real-locale dependence.
  - **Remove the static `DEFAULT_DAY` constant** (it's currently imported by cli.ts:22 and
    interpolated into help text at module-load, which would freeze the *dev machine's* default
    into the published help). Call sites call `defaultTriggerDay()` at runtime instead.
    `DEFAULT_TIME` (18:00) stays a constant.
- **Help text** for `--day` becomes static: `"Day of week for the weekly trigger (default:
  detected from your system timezone)"` — NOT the interpolated day.
- **Announce line is emitted Node-side in cli.ts**, after resolving the day and before/after
  calling `enableSchedule` (which delegates its own output to the PowerShell `Write-Host` and
  can't carry this): `console.log("drain scheduled: Friday 18:00 (Mon–Fri work week detected
  from America/New_York) — override: glean schedule enable --day Thursday")`. The test for the
  announce asserts on cli.ts behavior / the resolved day, not on PowerShell output.
- Precedence unchanged from v0.8.0's D7 fix: `--day` flag > `config.drain_trigger.day` > `defaultTriggerDay()`.
- The repetition window (`duration_hours` default 60) already comfortably spans Thu/Fri-evening → the next weekly reset; no change.

### Component 2 — shareable RECEIPT.md
- A new pure renderer `renderReceiptMarkdown(report: MorningReport): string` in a sibling
  `render-receipt.ts`, emitting markdown from the SAME `MorningReport` the terminal renderer
  builds — single data model, no second source of truth. The window aggregation
  (`getRunsWithCandidatesSince`) and the single-run fallback ALREADY happen inside
  `findMorningRun` (morning.ts), so the report handed to the markdown renderer already has the
  whole-window data by construction — zero new aggregation.
- **Reuse, don't re-code:** export `PLAIN` and `outcomeLine` from `render-morning.ts` and call
  them from `render-receipt.ts` (the honesty switch lives in ONE place — only `weekly-drained`
  may say "drained weekly capacity"). Do not duplicate the outcome switch.
- Layout:
  - **Header:** `# glean — <project> — <date / drain-window range>` + the `outcomeLine` capacity phrase.
  - **Totals line:** `N draft branches · M dossiers · +X / -Y across the window · ~Z min`.
    `Z` = summed per-burst active minutes (sum of each window run's `ended_at - started_at`),
    which is more honest than the wall-clock window span; fall back to wall-clock if a burst's
    `ended_at` is null. (Summing needs the per-run rows — surface a `bursts`/minutes total on
    `MorningReport`, or compute in morning.ts where the run rows are already in hand.)
  - **Per-item detail:** branch entries (diff stat, test status, review/discard commands, the
    "main never touched" trust line) and dossier entries — same content as the terminal.
- **WRITE PATH (resolved): a single `writeReceipt(gleanRoot)` helper writes it; called from ONE
  place — the end of `runPipeline`/`runDrain`, best-effort + silent-degrade** (like the existing
  morning/peek error handling). It calls `findMorningRun(gleanRoot)` (which aggregates the
  window + derives the project) → `renderReceiptMarkdown` → writes to
  `~/glean/dossiers/<proj>/<today>/RECEIPT.md`. This means every run refreshes the receipt with
  the whole-window view WITHOUT the user needing to run `glean morning` separately. `glean
  morning` then (a) prints the terminal receipt as today and (b) notes the RECEIPT.md path IF
  it exists. `glean morning --md` re-renders and prints the markdown to stdout on demand.
- **`--md` flag:** add `md: { type: 'boolean', default: false }` to `morningCmd`. When set,
  `glean morning --md` calls `renderReceiptMarkdown(findMorningRun(...))` and writes it to
  stdout with `useColor=false` semantics (no ANSI) — for piping into a PR/issue/clipboard.
- **Zero-burst / no-run cases:** if `findMorningRun` returns null (no run) → no RECEIPT.md
  written, and `glean morning` prints its existing friendly "no recent run" line (no false
  "receipt saved"). A 0-burst drain window writes a RECEIPT.md that says "0 bursts — nothing
  ran this window" (reusing the v0.8.0 honest 0-burst path), never a success.

### Component 3 — README refresh (sequenced last)
Not "a few line edits" — a substantive rewrite of ~5 major sections to v0.8.0 reality:
- **Install:** lead with `npm i -g @jonny-boy9000/glean`; keep from-source as secondary.
- **Hero/tagline:** keep the morning-receipt image; update the value line to mention the drain
  ("drains your weekend's leftover capacity") and the new scheduling.
- **Commands section (new):** `glean run [--drain]`, `glean schedule enable|disable|status`,
  `glean morning [--md]`, `glean today`, `glean rate`, `glean gc`, `glean stop`.
- **How it works:** add the drain core (exit-and-re-enter + Task Scheduler), draft-impl, and
  the rate-limit horizon classifier.
- **Fix the contradicted FAQs:** "burn my whole limit?" (now: `--drain` pauses/resumes across
  5-hour windows, stops at the weekly cap), "touch my main?" (draft-impl shipped, worktree-isolated),
  "run on a schedule?" (yes — `glean schedule enable`, work-week-aware default).
- **Output tree:** add `state/budget.json`, `work/<...>` prep worktrees, `RECEIPT.md`.
- **Advanced configuration:** the `base_branch` field is described as "currently a no-op" — it
  has been **load-bearing since v0.7.0** (enables draft-impl). Document `base_branch` +
  `test_command` + the new `drain_trigger` config properly.
- **Coming next:** replace done items with the real v0.8.1+ list (circuit-breaker tuning,
  mid-weekend re-discovery, anti-spill margin, today/peek window views, POSIX port) + the
  "one real overnight drain run" validation gate.
- **Scripting/exit-code table:** note the drain reasons (session-paused/weekly-drained/etc.)
  map onto existing exit codes.

## Spec Review (adversarial, 1 round)

Independent fresh-context review against the code, scored 7/10. Confirmed the design is sound
(PLAIN painter exists, `MorningReport` is the right single data model, `defaultTriggerDay(tz)`
is cleanly testable, bare `run`/`today` provably untouched). Two blocking gaps + several
specificity gaps, all now folded in:
- **RECEIPT.md write path** was ambiguous (who writes it, when, from where) → resolved: one
  `writeReceipt(gleanRoot)` helper, called best-effort at the end of every `runPipeline`/`runDrain`,
  so a receipt always exists without a separate `glean morning` invocation (Component 2).
- **`DEFAULT_DAY` going dynamic** would freeze the dev machine's day into the published help
  text → resolved: remove the constant, runtime `defaultTriggerDay()`, static help text, Node-side
  announce line (Component 1).
- Folded: export `PLAIN`/`outcomeLine` (no honesty-switch drift); `--md` flag spec; totals =
  summed per-burst minutes; zero-burst/no-run write behavior; `base_branch` README staleness.

## Open Questions
- **Which timezones besides `Asia/Jerusalem` map to Thursday?** Most former Sun–Thu countries
  moved to Fri–Sat or Sat–Sun weekends; start with Jerusalem only and expand if a user asks.
  (Don't over-build the table.)
- **RECEIPT.md path:** per-date dossier dir vs a top-level `~/glean/RECEIPT-latest.md` for a
  stable "always the newest" path. Lean: write per-window in the dossier dir AND keep a
  `morning` pointer; a stable-latest symlink/copy is a nice-to-have.
- **`glean today` header/totals:** Component 2 focuses on the morning receipt; applying the
  same header/totals polish to `glean today` is a cheap follow-on but not required.

## Success Criteria
- `glean schedule enable` on an `Asia/Jerusalem` machine schedules **Thursday** 18:00 and says
  so; on a Mon–Fri-zone machine schedules **Friday** 18:00; `--day` / config override both win;
  the resolved choice is printed. Unit-tested with an injected timezone (no real-clock/locale dependence).
- After a run/drain window, a `RECEIPT.md` exists with header + totals + per-item detail;
  it renders correctly on GitHub (paste test); totals match the terminal receipt; a 0-burst
  window and a weekly-drained window both read honestly. `glean morning --md` prints it.
- The bare `glean run` / `glean today` terminal output is unchanged (regression).
- README: install leads with npm; no FAQ contradicts shipped behavior; `glean schedule` +
  `--drain` + `glean morning` documented; "Coming next" contains no already-shipped item.
