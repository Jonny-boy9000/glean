## Load-bearing constraints (from CLAUDE.md — non-negotiable)
Subscription auth only (no API key in the core path); read-only against the user's `main`
checkout (drafts go to `prep/glean-*` worktrees, never push/merge); deny-list on **every**
`claude -p` spawn; default `max_parallel=1`; Windows-first; cross-invocation drain state lives
in a single atomic top-level `state/budget.json`.

## v0.8.2 scope — four robustness items + one validation gate

### 1. Configurable circuit-breaker threshold + richer "unproductive" classification
- **Now (v0.8.0):** `runDrain.ts` has a minimal hard backstop — `MAX_UNPRODUCTIVE = 3`
  consecutive **zero-task** bursts → stop the window with reason `no-progress`. A session/weekly/
  ambiguous rate-limit pause does NOT count as unproductive (only a burst that ran 0 tasks AND got
  no rate-limit signal does).
- **v0.8.2:** make the threshold configurable (a `config.drain_trigger` / new field) and upgrade
  "zero new tasks" → a richer signal that also trips on "ran tasks but all empty/failed/instantly
  rate-limited." `DrainState` already carries `unproductive_reentries` + `consecutive_ambiguous`.
- Files: `runDrain.ts`, `state.ts` (DrainState), `config.ts` + `types.ts`.

### 2. First-class mid-weekend candidate re-discovery
- **Now:** each burst re-runs discovery (so new TODOs/PRs CAN appear), and the resume cursor
  skips already-done work via the **stable `evidence_hash`** set (not random candidate ids).
- **v0.8.2:** make re-discovery deliberate + correct over a multi-day window — merge candidates
  that appeared after the window started, dedupe against `completed_task_ids` (evidence_hash), and
  handle files that **changed** under an in-flight draft. Confirm a multi-day drain isn't silently
  working off the first burst's snapshot.
- Files: `pipeline.ts` (discovery), `runDrain.ts` (skip-set), `dedup.ts`.

### 3. Anti-spill pre-emptive margin (the "first-task-blind" fix)
- **Now:** the drain stops *reactively* when the weekly-limit stderr fires. A task can still START
  seconds before the weekly reset and run a little into the fresh new-week allowance.
- **v0.8.2:** refuse to **start** a new task within a configurable margin (default ~15 min) of a
  **known** `last_observed_weekly_reset` estimate. `DrainState.last_observed_weekly_reset` already
  exists. Note the genuine blind spot: on the very FIRST burst of a fresh window there's no reset
  estimate yet — document that it stays reactive there.
- Files: `runDrain.ts` guard / the pipeline execute loop, `state.ts`, config (margin).

### 4. `today`/`peek` window-aware aggregation (eng-review decision D7)
- **Now:** `morning` aggregates the whole drain window (`getRunsWithCandidatesSince`), but
  `today`/`peek` are still single-run / single-day → inconsistent during a multi-day drain
  ("morning says 4 branches, today says 1").
- **v0.8.2:** make `today.ts`/`peek.ts` window-aware to match `morning` (reuse the same
  `getRunsWithCandidatesSince` + `readDrainState` path), preserving their existing single-run
  behavior when no drain window is active.
- Files: `today.ts`, `peek.ts`, `morning.ts`/`memory.ts` (reuse).

### 5. VALIDATION GATE (highest value) — one real overnight/weekend drain run
- The classifier (`classify.ts`) is **built against plausible-but-unverified rate-limit stderr
  strings** (Spike 0 confirmed there is no headless `claude usage` query, so the stderr classifier
  is load-bearing; the real wording was never captured). Run a real drain against a live project,
  capture the ACTUAL session + weekly rate-limit stderr from `~/glean/logs/<run>/<task>.stderr`,
  and replace `classify.ts`'s format-table strings with verified ones (+ fixtures). Until this is
  done, the horizon classifier is the single biggest correctness risk in the drain.
