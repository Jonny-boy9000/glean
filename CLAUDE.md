## What `glean` is

A local CLI (`glean`) that consumes idle Claude Pro/Max subscription capacity at the tail of the weekly rate-limit window. It scans the user's `~/.claude/projects/*.jsonl` session history and their git repos for unfinished work, then spawns headless `claude -p` sessions to produce speculative drafts, research dossiers, and pre-fetched docs into `~/glean/`. The next session, the user opens a "prep folder" and finds a head-start.

**The "free idle capacity" thesis is true but CONDITIONAL** (audit #12, HOLDS-scoped). It assumes the user has an actual idle *tail* on their **shared** weekly pool (Claude Code + claude.ai chat + Cowork all draw the same cap), measured by `glean usage`. A heavy week may leave no tail, in which case the drain competes with the user's own real sessions — the pacing engine gates this *by design* (`recommendTier()` returns `skip` when not underspending). It is also not strictly "use-it-or-lose-it" for everyone: Pro/Max users can opt into **extra usage** (Settings → Usage, off by default, billed at API rates with a spend cap), so for a user who enabled that toggle, draining past the included limit can cost real money. State the thesis in the conditional; point users at `glean usage` before `glean schedule enable`; never re-assert "the only alternative is losing it" (verified live 2026-06-23 — see the audit's #12 and the watchlist).

It is **not** an API-token reseller or marketplace — that alternative was explicitly dropped (see `glean.md` §2) because Anthropic's terms prohibit reselling/proxying Claude access.

## Decision records & assumptions

Load-bearing decisions — and especially **unverified assumptions** — are recorded as tiny ADRs in
[`docs/decisions/`](./docs/decisions/) and tagged at the code site (`ASSUMPTION[ADR-NNNN]` /
`INVARIANT[ADR-NNNN]`). This exists because rationale stated as *fact* (in a comment or memory)
once led a later session to "correct" a subsystem wrongly — the fix is to mark **verified vs.
assumed, loudly, where the code is**. The decision record currently runs **ADR-0001 → ADR-0012**;
the [2026-06-23 assumption audit](./docs/strategy/2026-06-23-assumption-audit.md) is the companion
register that ranks how solid each assumption is and queues the next ADRs. Rules:

- Touching a subsystem with an `ASSUMPTION[ADR-NNNN]` tag? **Read that ADR first.** It tells you
  what's a guess and what would change it.
- Making a load-bearing or unverified decision, or reversing one? **Add/supersede an ADR** (never
  edit an old one; supersede it) and tag the code site. See `docs/decisions/README.md`.
- **A finding that overturns a prior decision is a hypothesis to disprove, not a conclusion** —
  verify the negative case before asserting (evidence before assertions). The rate-limit signal
  (ADR-0001) is the worked example: a "real signal found!" claim was a warning, not the block. The
  audit's #9 cross-check is the second worked example: a sub-auditor's "supersede to Opus-only" was
  *disproved*, not adopted — inverting it would have re-introduced the ADR-0001 failure mode in reverse.
- **Any EXTERNAL fact** (Anthropic ToS, billing, sandbox/permission semantics, CLI behavior, plan
  economics) must be **web-verified against a primary source with a date**, not asserted from model
  memory — the knowledge cutoff is January 2026. The audit verified its external facts live on
  2026-06-22/23; the watchlist below records the canonical URLs and the re-check cadence.

## Load-bearing constraints (do not violate)

- **Subscription auth, no API key.** Capacity is consumed by spawning `claude -p "<prompt>"` subprocesses. Never introduce direct Anthropic API calls or `ANTHROPIC_API_KEY` usage in the core path — the whole tool assumes Pro/Max session auth. **The one sanctioned exception is NOT an API key:** scheduled/drain runs *may* authenticate via `claude setup-token` → `CLAUDE_CODE_OAUTH_TOKEN` (subscription auth, 1-year, inference-only, draws from the **same** weekly pool — verified `code.claude.com/docs/en/authentication`, 2026-06-23). It is the cause-fix sibling of the ADR-0010 auth-failure *detection* and is **designed, not yet built** (the print-mode OAuth-non-refresh bug makes a mid-drain 401 plausible on a multi-hour unattended drain). When built, it injects the token into the child env **only on `--drain` runs**, stores it `0600` (never on the Task Scheduler command line, never in `config.json`), and **strips** `ANTHROPIC_API_KEY`/`ANTHROPIC_AUTH_TOKEN` from that env (precedence is API_KEY > OAUTH_TOKEN > /login). Never pass `--bare` (it ignores the OAuth token). Interactive `glean run` is unchanged.
- **Read-only against the user's primary checkouts.** All speculative output goes to `git worktree`-based `prep/*` branches under `~/glean/work/` or to dossier dirs under `~/glean/dossiers/`. Never mutate the user's main checkout or push anything. **Boundary model (ADR-0009 + audit #1/#2):** built-in Edit/Write are bounded to the worktree via `--add-dir`, and the default-deny allow-list grants no bare `Bash`; but an allow-listed *interpreter* verb (`node`, `npm run`, a test runner) spawns a subprocess **outside** the permission layer, and the only real filesystem boundary is the OS sandbox, which exists **only on macOS / Linux / WSL2** (Seatbelt / bubblewrap), not on native Windows. **The one hard guarantee that exists TODAY on every platform (incl. native Windows) is `config.strict_spawn: true`, which drops in-session code execution entirely (the OS sandbox would restore it safely on macOS/Linux/WSL2 — queued).** Three postures govern the spawn:
  - **Narrow (default).** The allow-list **excludes** `node`/`npm run`; in-session subprocess writes are **defense-in-depth, not a hard guarantee** (this is the current state on every platform). Verified against Anthropic's own permission docs: deny rules bound tool-call *names*, not what an allow-listed interpreter then does.
  - **`strict_spawn: true` (opt-in).** Drops in-session code execution entirely — a **platform-independent hard "no in-session code"** guarantee, and the ONLY hard guarantee available right now on native Windows.
  - **`enforce_spawn` (queued, supported platforms only).** Inject Claude Code's OS sandbox per-spawn via inline `--settings` (never mutating the user's `~/.claude/settings.json`), scoping writes to the worktree/dossier and denying read of `~/` secrets + the main checkout, with `sandbox.failIfUnavailable: true` so it **fails closed** rather than running unsandboxed. This is the only thing that makes in-session subprocess writes a **HARD OS boundary** — and only on macOS/Linux/WSL2; on native Windows it degrades to Narrow with a **loud receipt warning**, never silently. Where the sandbox is on, glean may safely restore the `node`/`npm run` test verbs. **Plainly: in-session subprocess writes are a HARD OS boundary only where the sandbox runs (mac/Linux/WSL2); on native Windows they are defense-in-depth only, unless `strict_spawn` is set (which removes them entirely).**
- **Spawned sessions must run with a deny-list AND a scoped default-deny allow-list.** Every `claude -p` invocation passes `--disallowedTools "Bash(git push:*) Bash(git checkout main:*) Bash(gh pr merge:*) Bash(gh pr create:*)"`; `draft-impl`/`research-dossier` also pass a scoped `--allowedTools` (never bare `Bash`). Do not remove these even if a task seems to need them. The deny-list is in-session defense-in-depth; the allow-list (+ `--add-dir`) bounds which tools/writes are possible at all (ADR-0009). A real enforcement test against the live `claude` binary (asserting an out-of-worktree `fs.writeFileSync` and a worktree pre-commit hook are both refused) is **queued** — until it runs on a sandbox-capable box, the argv tests prove *shape only*, not the boundary.
- **Privacy / data-minimization & read-scope confinement (queued; audit #6).** glean reads **full** `~/.claude` transcripts (the JSONL signal is core to discovery) — which can hold pasted secrets/PII — and derived titles + TODO excerpts land **at rest** in `memory.db`, `~/glean` dossiers/`INDEX.md`, and rendered prompts. The standing policy: (a) confine every spawn's `--add-dir` read-scope to the candidate's **own** `project_path` — never another repo — as an enforced, tested invariant (today the only granted dir *is* `project_path`; keep it that way, and do not widen it for a future multi-project/PR-path candidate without the guard); and (b) best-effort-redact obvious secrets from transcript/excerpt-derived text before persistence. Redaction is **high-precision/low-recall defense-in-depth, NOT a guarantee** (raw spawn logs under `~/glean/logs/**` stay unredacted). Do not persist raw transcript text without routing it through the redaction pass, and do not grant a spawn read-scope beyond its candidate's repo.
- **No cross-invocation prompt caching glean can exploit.** Subscription auth can't manage `cache_control` by hand; Claude Code auto-caches (auto ~1h TTL on subscription), but reuse is keyed on the exact prefix *including cwd*, and glean's per-worktree / per-dossier dirs diverge — so natural cross-invocation cache share is ~zero. Don't design around a pre-warmed cache.
- **Rate-limit budget is indirect.** There is no separate headless `claude usage` probe (Spike 0). The executor reacts to a wall-clock `--budget`, a `~/glean/STOP` sentinel, and a rate-limit signal. **Correction (2026-06-02):** the real `claude -p` rate-limit signal is **not** stderr prose — it is a structured `{"type":"rate_limit_event","rate_limit_info":{…}}` message in the `--output-format stream-json` stream glean already captures to `~/glean/logs/<run>/<task>.jsonl` (carries `rateLimitType` e.g. `five_hour`, `resetsAt` epoch-seconds, `status` incl. a proactive `allowed_warning` + `utilization`). Classify off that; keep the stderr regex as a fallback. **Weekly horizon (ADR-0003, audit #11):** classify the weekly block FIRST-CLASS off `rateLimitType` — any `seven_day*` bucket (`seven_day`/`seven_day_opus`/`seven_day_sonnet`) is weekly regardless of the 6-hour `resetsAt` cut, which is retained only as the label-absent/`five_hour` fallback. The weekly BLOCK shape itself is **still unverified** (no real weekly block has ever been captured); the BLOCK-CAPTURE tripwire stays armed. A continuous server-truthful weekly utilization is **not** available to a headless `-p` spawn (verified 2026-06-23: the richer `rate_limits.seven_day` object is statusLine-TUI-only); local-JSONL stays the continuous primary, with any opportunistically-observed `seven_day*` utilization fused as a **caution-only ceiling**. The local-JSONL `haircut` is a fallback for the off-machine blind spot, **not** a fix for it. See [`docs/open-work/06-rate-limit-signal-findings.md`](./docs/open-work/06-rate-limit-signal-findings.md). Still don't invent a token-counter abstraction.
- **Default AND only supported value is `max_parallel=1` (hardcoded — a SAFETY default, not merely a rate-budget choice).** Subscription sessions share one rate-limit bucket, so parallelism mostly accelerates exhaustion; additionally, concurrent same-project `draft-impl` is **UNSAFE** — two spawns race on the shared `.git` (`worktree add` / `branch -D` / `prune` / per-worktree `index.lock` are not cross-process-coordinated). `--parallel` is intentionally **NOT exposed** (any doc that says "`--parallel 2` is exposed" is stale doc-drift — the flag is not wired; the pipeline loop is strictly serial). A per-project worktree-mutation lock is the prerequisite to ever exposing it (queued; audit #10).

## Architecture pointers (read `glean.md` for detail)

- **§5.1 Discovery** — parallel read-only passes: JSONL session signals, git scan (`git grep TODO/FIXME`, stale `claude/*` branches, `gh pr list`), `find -mtime -7`, plus the v0.9 `discover-docs` pass mining the project's own planning docs (ROADMAP/TODO/handoff "up next" items) as candidates. Emits `candidates.json`.
- **§5.2 Prioritizer** — ranks by `est_value / log(est_tokens + 1)` with type weights (`draft-impl` 1.0 → `fetch-docs` 0.2). Because `est_tokens` is a coarse per-source constant (see Resolved decisions), the denominator is near-inert within a type; ranking effectively collapses onto `TYPE_WEIGHT * est_value * pathPenalty`. Last 30 min of budget restricts to `fetch-docs` only.
- **§5.3 Executor** — per-candidate: provision workspace (worktree for `draft-impl`, dossier dir otherwise), render template, spawn `claude -p` with `--output-format stream-json --include-partial-messages` + deny-list + scoped allow-list + 8 min timeout, capture output to `OUT.md` (or auto-commit for `draft-impl`). The worktree's hooks are neutered (`core.hooksPath`) before the spawn; glean's own out-of-session test re-run (`draft-test.ts`) runs **after** the spawn process tree is dead.
- **§5.4 Dossier layout** — `~/glean/{dossiers,work,logs,state,templates}/...`. `INDEX.md` per date sorts by realized value and includes Apply/Discard hints.
- **§5.5 Scheduling** — `glean schedule enable` registers a Windows Task Scheduler task (launchd plist on macOS / crontab line on Linux); off by default. The unattended/scheduled path is the gray edge of the ToS basis (see below). Optional SessionStart hook calls `glean peek`.
- **§5.6 Safety** — `glean discard`, `glean gc` (21-day worktree expiry), STOP sentinel checked between every task, the ADR-0009 spawn boundary, and the ADR-0010 auth-error stop.

## ToS basis (load-bearing, gray-but-defensible; audit #3)

Scheduled headless `claude -p` is **permitted on a defensible-but-gray basis**, not an unconditional "Yes":
- It drives the **official** Anthropic binary `claude -p` — the textbook "where we otherwise explicitly permit it" case under Consumer-Terms §3 (Anthropic's own docs promote `tail -f log | claude -p` and ship Claude Code GitHub Actions on cron).
- It is **NOT** the banned OpenClaw class — no OAuth-token extraction/reuse outside the official binary, no proxying, no shared accounts, no API-key reselling.
- The **untested edge is unattended/scheduled vs. interactive** use: §3 spells out neither, and the written terms do not textually carve out official-binary automation, so this leg is **UNVERIFIED**. The metered-billing un-pause is the existential tripwire (ADR-0008 hedge; watched weekly).

The ToS-basis ADR ([ADR-0011](./docs/decisions/0011-tos-basis-for-scheduled-claude-p.md)) + the `ASSUMPTION[ADR-0011] UNVERIFIED` tags at the unattended-execution sites (`runDrain.ts`, `schedule.ts`) + the `docs/watchlist/tos-automation-drift.md` watch **shipped (Phase 0, 2026-06-23)**. The user-facing README answer is dated/conditional (PR #31).

## MVP scope (§6)

Smallest end-to-end slice — build this first, do not pre-optimize:
- Manual trigger only, single project (`--project <path>`), 60-min budget.
- Discovery: JSONL last-`aiTitle` + `git grep TODO/FIXME` + `gh pr list`. Skip the other signals for now.
- Candidate types: `research-dossier` and `fetch-docs` only. Skip `draft-impl` worktrees in MVP.
- Serial executor, 8-min per-task timeout, clean exit on rate-limit stderr.
- Manual discard via `rm -rf`. No schedule, no hooks.

Target: ~600–900 LOC.

## Development substrate decision (§10)

**Build bare on Claude Code CLI primitives. Do not introduce Ruflo as a dependency** even though it is installed on this machine — `glean`'s runtime substrate *is* `claude -p` + git + shell, so wrapping it in Ruflo's router would abstract away the exact primitives the tool ships on top of, and would force `glean` users to install Ruflo too.

Ruflo *may* show up later as an **output integration** (writing dossiers into AgentDB so future Ruflo sessions can search them), but never as a dev dependency for the tool itself.

When developing here:
- Use the Claude Code CLI in this repo. Avoid the desktop app — the CLI exposes the shell environment the tool targets.
- Use the `loop` skill for build → test → iterate cycles during *development*, not as the tool's runtime.
- Use `git worktree` for parallel experiments — same primitive the tool will ship.
- No plugin framework; a ~900 LOC tool doesn't need one.

## Resolved decisions (override conflicting spec language)

See `glean.md` §0. Summary:

- **Windows-first.** Development happens on Windows 11; macOS/Linux is post-MVP (Linux beta shipped v0.8.4). Replace bash scripts with Node/TS; schedule via Windows Task Scheduler, not launchd/cron. (Note: the OS sandbox that would make the spawn boundary *hard* exists only on macOS/Linux/WSL2 — see the read-only constraint — so on the primary dev platform the boundary stays defense-in-depth, and `strict_spawn` is the only hard guarantee.)
- **Node + TypeScript.** `bin/glean` is a Node CLI; all `lib/*` are `.ts`. Shell-out only for `git` and `gh`.
- **Base branch is explicit per-project** in `%USERPROFILE%\glean\config.json` (`project_path → base_branch`). No autodetection in MVP; missing entry → skip `draft-impl` for that project with a warning.
- **Heuristic-only ranking for MVP.** `est_value` from discovery-evidence signal strength; `est_tokens` is a **coarse per-source constant** (≈2000–6000 by discovery source, hardcoded in `discover-*.ts` — *not* a per-candidate template/excerpt size estimate, and **not** the "×1.3" the old docs wrongly claimed), so the `log(est_tokens+1)` denominator is near-inert within a type. No upfront `claude -p` triage pass unless rankings prove bad post-MVP — a cheap NON-LLM **validity guard** (drop stale candidates before ranking) and an opt-in, single-batched-spawn `--triage` pass are the sanctioned next steps and are **queued** by the audit (#7).

## Verification checklist (§8)

When something is implemented, the spec lists explicit acceptance tests: dry-run discovery, single-task execution under a 15-min budget, budget self-termination (`summary.json.reason == "wall-clock"`), STOP sentinel, rate-limit back-off via a mocked `claude` stub, and `glean discard` reversibility. Use these as the test plan rather than inventing new ones. **Honesty rule (from the audit):** a green suite that **skipped** a gated proof (e.g. the real spawn-enforcement test that only runs on a sandbox-capable box) is NOT a green suite that **proved** it — surface the skip so the distinction is visible, and keep `exit 0 → pass` the only fully-trustworthy draft test signal.
